Architecture & Design

How each system is built, what patterns it uses, and how cleanly the architecture supports its goals.

AspectAgentOpsOpen Brain (OB1)
Core Pattern Strong
Event-driven hook system. Hooks intercept agent actions (PreToolUse, PostToolUse), run checks, emit events to an event bus. Plugins subscribe to events.		 Strong
MCP tool server. Four registered tools on a Hono HTTP server. Stateless request/response via Streamable HTTP Transport.
Deployment Model Strong
Local-first. Directory of scripts, configs, and HTML in your project root. No server, no cloud, no dependencies beyond shell and git.		 Moderate
Cloud-required. Supabase Edge Function + PostgreSQL. Requires internet, Supabase account, and OpenRouter API key.
Data Storage Moderate
Flat files (scaffold docs, JSON logs, audit trail). No database. Simple but limits queryability.		 Strong
PostgreSQL with pgvector. Structured data + vector embeddings. Full SQL queryability + semantic search.
Codebase Size Strong
Shell scripts averaging 50-100 lines each. HTML dashboard ~800 lines. Auditable by non-engineers.		 Strong
~250-line core server. Extensions are self-contained. Entire server fits in one file.
Modularity Strong
5 skills are independent modules. Event bus decouples plugins. Each script is standalone. Clean separation of concerns.		 Moderate
4 tools in a single server file. Extensions are separate. No formal module boundary within the core — works at this scale, might not at 10x.
Scalability Path Strong
Plugin architecture + event bus designed for growth. Tracing (OpenTelemetry), agent identity (RBAC), cost management, lifecycle management all specified.		 Moderate
Supabase handles database scaling. Adding tools means growing the server file. No formal plugin/hook system for core extensions.
Offline Capability Strong
Fully offline. Everything runs locally. No API calls required for any core feature.		 Limited
Requires internet for Supabase, OpenRouter embeddings, and LLM metadata extraction. No offline mode.
State Management Moderate
Scaffold docs (markdown files) + JSON config. Human-readable but no transactional guarantees. Concurrent agent writes could conflict.		 Strong
PostgreSQL ACID transactions. Row-level security for multi-user isolation. Concurrent writes handled by the database.
Observability Strong
OpenTelemetry-compatible tracing specified. Dashboard with health rings, KPIs, trend charts. 8-page dashboard prototype built.		 Moderate
thought_stats tool gives aggregate counts. No tracing, no dashboard in core (community-contributed dashboards exist as a category).
Bottom line: AgentOps has the more ambitious, enterprise-grade architecture (event bus, plugins, tracing, RBAC). OB1 has the more elegant, minimalist architecture (250 lines, one database, four tools). Neither is "better" — they're right-sized for their problem domains.

Security Model

How each system protects the user, the project, and the data.

AspectAgentOpsOpen Brain (OB1)
Data Residency Strong
All data stays local. Files in your project directory. Nothing leaves your machine. Full data sovereignty.		 Moderate
Data lives in Supabase cloud (user's project, but hosted). Thoughts sent to OpenRouter for embedding. Cloud dependency.
Secret Detection Strong
Dedicated secret scanner. Pre-commit hook blocks secrets. Regex patterns for API keys, tokens, connection strings, AWS keys, etc.		 Moderate
Automated PR check rejects contributions containing credentials. No runtime secret scanning on captured thoughts.
Access Control Strong
3-layer RBAC specified: agent identity, capability model, permission boundaries. Delegation tokens for agent-to-agent trust.		 Strong
Row-level security via PostgreSQL RLS. MCP access key for server authentication. Multi-user isolation built in.
Audit Trail Strong
Append-only, SHA-256 hash-chained audit records. Tamper detection. EU AI Act Article 12 compliance specified.		 Limited
No formal audit trail. Thoughts have timestamps and metadata, but no immutability guarantees or tamper detection.
Input Validation Strong
Input validation at all system boundaries is a core design principle. Rules file linting. Path sanitization.		 Moderate
Zod schema validation on MCP tool inputs. SQL safety checks (no DROP, TRUNCATE). Core table protection rules.
PII Protection Strong
Dedicated PII detection module. Scans for emails, phone numbers, SSNs in code. Blocks commits containing PII.		 Limited
No PII detection. Thoughts can contain any content. User is responsible for what they capture.
Blast Radius Control Strong
Risk scoring model (file count + DB changes + shared code). Auto-branching for high-risk operations. Decomposition enforcement.		 N/A
Not applicable — OB1 doesn't modify project files. It only reads and writes to its own database.
Compliance Readiness Strong
EU AI Act, SOC 2, hash-chained audit trails, compliance report generator. Enterprise-ready compliance framework.		 Limited
FSL-1.1-MIT license. No compliance tooling. Not designed for regulated environments.
Supply Chain Safety Moderate
Plugin validation specified (no secrets, metadata checks). But no plugin signing or dependency scanning.		 Strong
11 automated PR checks. No binaries >1MB. No destructive SQL. Metadata schema validation. Human review gate.
Bottom line: AgentOps wins decisively on security. It's fundamentally a safety system — secret scanning, PII detection, blast radius control, and compliance auditing are its raison d'être. OB1 has sensible guardrails for a knowledge tool but isn't built to be a security product.

Memory, Data & Cross-Session Intelligence

How each system persists knowledge across sessions and makes historical data useful.

AspectAgentOpsOpen Brain (OB1)
Cross-Session Memory Moderate
Scaffold documents (PLANNING.md, TASKS.md, CONTEXT.md, WORKFLOW.md) carry state between sessions. Human-readable but lossy — history is overwritten on each update.		 Strong
Every thought persists with full history. Nothing is overwritten. Vector embeddings enable retrieval by meaning across unlimited sessions.
Semantic Search Absent
No semantic search. Scaffold docs are text files. Finding past decisions requires manual reading or grep.		 Strong
Core feature. pgvector embeddings with configurable similarity thresholds. Search by meaning, not keywords.
Auto-Classification Moderate
Events are classified by the module that generates them (hardcoded categories). No cross-cutting classification.		 Strong
LLM-powered auto-extraction: type, topics, people, action items, dates. Zero manual tagging required.
Knowledge Compounding Limited
Each session starts semi-fresh. Scaffold docs provide context but don't learn from patterns across sessions.		 Strong
Core design principle. Extensions compound (CRM references captured thoughts; meal planning checks household schedules). Data flows between systems.
Data Portability Strong
Everything is markdown and JSON files in your project. Copy the directory, you have all your data. Zero lock-in.		 Moderate
Data in Supabase PostgreSQL — exportable via SQL dump or API. But embeddings are tied to the embedding model used.
Multi-Tool Data Access Moderate
Scaffold docs are readable by any tool (they're markdown). But structured data (audit logs, events) is tool-specific.		 Strong
Any MCP-compatible client (Claude, ChatGPT, Cursor) accesses the same database. True multi-tool parity for read and write.
Historical Trends Strong
Dashboard with trend charts, historical KPIs, agent activity timelines. Designed for operational trend analysis.		 Moderate
thought_stats gives aggregate counts by type, topic, person. No time-series visualization in core.
Bottom line: OB1 wins on memory and cross-session intelligence. It's built specifically for this. AgentOps treats memory as a secondary concern (scaffold docs are a state handoff mechanism, not a knowledge system). This is the #1 area where AgentOps should adopt OB1's patterns — see the cross-pollination analysis.

Choose AgentOps When...

You're managing AI coding agents on real projects and need to prevent disasters — data loss, security gaps, context drift, blast radius problems. You want enterprise-grade safety rails. You care about compliance and audit trails. You need multi-agent coordination. You want offline, local-first, zero-dependency infrastructure.

Choose OB1 When...

You use multiple AI tools and your knowledge is scattered across them. You want every AI you use to share the same memory. You want to capture thoughts, decisions, and ideas once and retrieve them by meaning from any tool. You want a gentle learning curve and a community to contribute to.

The real answer: Use both. OB1 provides the persistent memory layer that AgentOps lacks. AgentOps provides the safety and management layer that OB1 lacks. The cross-pollination analysis identifies exactly how to integrate OB1's patterns (persistent memory, MCP interface, primitives, contribution model) into AgentOps — making both systems stronger.
